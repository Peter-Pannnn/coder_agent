# CoderAgent

一个面向代码仓库的 Agent 助手项目。它的目标不是替代开发者，而是帮助开发者更快理解项目结构、定位代码、解释实现逻辑、生成修改建议、辅助编写测试。

## 项目目标

CoderAgent 主要解决以下问题：

- 快速理解陌生代码仓库的目录结构、核心模块和技术栈。
- 根据自然语言问题定位相关文件、函数、类、配置和测试。
- 解释代码逻辑、调用链、数据流和模块依赖关系。
- 辅助生成修改方案、测试计划、风险分析和代码补丁。
- 支持仓库级别的问答，例如“配置加载在哪里实现？”、“这个接口的数据从哪里来？”、“如何添加一个新的配置项？”。

## 范围说明

第一阶段不考虑用户登录、账号体系、权限后台和多租户管理。项目优先聚焦本地代码仓库分析、检索、问答和变更计划。

## 典型使用场景

- 新人接手项目时，快速了解项目架构。
- 开发需求前，定位相关代码和影响范围。
- 修 Bug 时，追踪调用链和可能的根因。
- Code Review 前，解释变更逻辑并检查风险。
- 为已有代码补充单元测试或集成测试。
- 自动生成模块文档、接口说明和重构建议。

## 核心能力设计

### 1. 仓库扫描

Agent 首先需要读取仓库结构，识别项目类型和关键文件。

可扫描的信息包括：

- 文件目录树
- 语言和框架
- 依赖配置，例如 `package.json`、`pyproject.toml`、`pom.xml`、`go.mod`
- README、文档、配置文件
- 测试目录和测试框架
- 入口文件、路由文件、服务层、数据库模型

### 2. 代码索引

为了让 Agent 能准确回答仓库问题，需要构建代码索引。

推荐索引内容：

- 文件路径索引
- 函数、类、变量、接口索引
- import/export 依赖关系
- 函数调用关系
- 文档注释和 README 内容
- Git 历史和最近变更，可作为进阶能力

MVP 阶段可以先用文本检索和语义向量检索，后续再加入 AST 分析。

### 3. LangChain Agent 编排

项目使用 LangChain 作为 Agent 框架，负责意图识别、工具调用、上下文组织和模型交互。

LangChain 在本项目中的职责：

- 组织系统提示词和任务提示词。
- 根据用户问题选择合适工具。
- 调用代码搜索、文件读取、Chroma 检索、测试执行等工具。
- 将检索到的代码上下文交给模型。
- 输出解释、计划、风险分析或可应用的修改方案。

### 4. Chroma 向量检索

项目使用 Chroma 作为向量数据库，用于保存代码片段、文档片段和元数据。

推荐写入 Chroma 的内容：

- 文件路径
- 代码片段
- 函数或类名
- 语言类型
- 起止行号
- 最近更新时间
- chunk hash，用于判断是否需要重新索引

Chroma 检索结果必须保留来源信息，回答时需要引用具体文件路径，必要时引用行号。

### 5. 检索增强问答

用户提问后，系统先检索相关代码片段，再交给大模型回答。

基本流程：

1. 接收用户问题。
2. 分析问题意图：解释代码、定位文件、修改代码、生成测试、审查风险等。
3. 检索相关文件和代码片段。
4. 将检索结果和问题一起传给模型。
5. 输出带文件路径和代码引用的回答。

### 6. 工具调用

Agent 应该可以调用工具完成真实任务，而不仅仅是聊天。

推荐工具：

- `list_files`: 查看目录结构。
- `read_file`: 读取文件内容。
- `search_code`: 全局搜索关键字。
- `index_repository`: 构建代码仓库索引，支持目录或具体文件路径输入，并将代码片段写入 Chroma。
- `retrieve_context`: 从 Chroma 索引中检索相关代码上下文。
- `inspect_symbol`: 查询函数、类或变量定义。
- `run_tests`: 运行测试。
- `write_patch`: 生成或应用补丁。
- `explain_error`: 分析报错日志。

### 7. 修改建议与安全控制

代码仓库助手必须重视安全边界。

建议规则：

- 默认只读分析，不直接改文件。
- 修改前先展示计划和影响范围。
- 涉及删除、批量重构、依赖升级时必须人工确认。
- 修改后运行相关测试。
- 输出结果要标明置信度和潜在风险。

## 技术选型建议

### 后端

- Python：主要开发语言。
- LangChain：Agent 编排、工具调用和上下文组织。
- Typer：构建命令行入口。
- FastAPI：可作为后续 API 层，MVP 阶段不是必须。

### 检索与索引

- ripgrep：快速文本搜索。
- tree-sitter：解析多语言 AST。
- Chroma：向量数据库，用于代码片段和文档片段的语义检索。
- SQLite：存储文件索引、符号索引和任务历史。

### 暂不考虑前端

- 第一阶段不开发 Web 前端。
- 主要通过 CLI 使用。

### 命令行

- Typer：Python CLI。
- Rich：美化命令行输出和表格。

## MVP 功能范围

第一版建议只做这些功能：

- 扫描一个本地仓库。
- 展示项目目录树和技术栈摘要。
- 支持自然语言提问。
- 使用关键词检索 + Chroma 语义检索 + 文件读取回答问题。
- 提供构建索引工具 `index_repository`，支持将仓库代码写入 Chroma。
- 提供上下文检索工具 `retrieve_context`，支持从索引中召回相关代码片段。
- 回答时给出引用文件路径。
- 支持“解释某个文件/函数”。
- 支持“帮我找某个功能在哪里实现”。
- 只读模式，不自动修改代码。
- 提供一个明确的“生成修改计划”能力，但暂不自动应用修改。

## 进阶功能

后续可以逐步加入：

- 语义检索和向量索引。
- AST 符号级索引。
- 调用链分析。
- 自动生成测试。
- 自动生成变更计划。
- 代码补丁生成。
- PR Review Agent。
- Git commit 历史分析。
- 多 Agent 协作，例如 Research Agent、Patch Agent、Review Agent、Test Agent。

## Agent 工作流示例

### 查询代码位置

用户：

```text
配置加载在哪里实现？
```

Agent：

1. 搜索 `config`、`settings`、`env` 等关键词。
2. 读取配置模块、入口文件和相关测试文件。
3. 汇总配置加载流程。
4. 返回相关文件路径和核心函数。

### 解释代码逻辑

用户：

```text
解释 src/services/payment.ts 的主要逻辑。
```

Agent：

1. 读取目标文件。
2. 识别导出的类、函数和依赖。
3. 解释核心流程。
4. 标出外部依赖和潜在风险。

### 生成修改方案

用户：

```text
我想给订单接口增加优惠券字段，应该改哪些地方？
```

Agent：

1. 定位订单接口、模型、DTO、数据库字段、测试。
2. 分析影响范围。
3. 给出修改步骤。
4. 生成测试建议。
5. 等待用户确认后再生成或应用修改方案。

## 项目目录建议

```text
codebase-agent-assistant/
  src/
    models/                    # 核心数据模型和领域对象
    agents/                    # Agent 实现
    tools/                     # 工具系统
    context/                   # 上下文工程
    prompt/                    # 提示词
    storage/                   # 本地存储
    cli/                       # 命令行入口
  tests/
  examples/
```

目录说明：

- `models/` 放核心数据结构，例如仓库信息、检索结果、代码片段、变更计划。
- `agents/` 放 LangChain Agent 的具体实现，可以从单 Agent 起步，后续再拆出规划、审查、补丁 Agent。
- `tools/` 放 Agent 可调用工具，例如读文件、搜索代码、构建 Chroma 索引、检索上下文、运行测试、生成 patch。
- `context/` 放上下文工程逻辑，包括代码切块、检索结果排序、上下文压缩和短期记忆。
- `prompt/` 统一管理提示词，避免提示词散落在业务代码里。
- `storage/` 负责 SQLite、Chroma 等本地持久化。
- `cli/` 是第一阶段主要交互入口，暂不开发 Web 前端。

## 开发路线

### Phase 1: 可用原型

- 实现本地仓库扫描。
- 实现文件读取、代码搜索、构建索引和上下文检索工具。
- 使用 LangChain 接入大模型，支持基于检索结果回答问题。
- 使用 `index_repository` 构建 Chroma 代码索引，并使用 `retrieve_context` 检索代码片段。
- 输出引用文件路径。
- 提供 CLI 入口。

### Phase 2: 索引增强

- 完善 Chroma 向量检索和索引构建策略。
- 加入符号索引。
- 缓存扫描结果。
- 支持大型仓库的分块检索。

### Phase 3: 工程化能力

- 支持运行测试。
- 支持生成补丁。
- 加入人工确认机制。
- 加入日志、trace 和错误恢复。

### Phase 4: 团队能力

- 支持多仓库管理。
- 支持团队共享知识库。
- 支持 GitHub PR Review。
- 支持权限管理和审计日志。

## 成功标准

一个合格的代码仓库助手应该满足：

- 回答能引用具体文件路径。
- 不凭空编造不存在的函数或文件。
- 能说明自己的检索依据。
- 修改代码前能给出清晰计划。
- 修改后能给出测试结果或未测试原因。
- 对不确定的问题会主动说明不确定性。

## 后续可扩展方向

- GitHub App：自动审查 Pull Request。
- 企业知识库 Agent：结合代码、文档、Issue 和设计稿回答问题。
- 自动迁移助手：辅助框架升级、依赖升级和代码重构。
- Bug 定位 Agent：根据报错日志自动定位相关代码和可能根因。
